## Syne architecture (current)

Этот документ описывает **текущее состояние Go-core** в репозитории (CLI — это только оболочка для ручного тестирования).

### Цель и ограничения

- **Цель**: обмен сообщениями в локальной сети без внешнего интернета.
- **Ограничения текущего MVP**:
  - TCP transport без устойчивых соединений (dial → write → close).
  - Никакой криптографии/подписей на сообщениях (пакет `core/crypto` пока заглушка).
  - Нет настоящих групп/серверов/relay (только заготовки).

### Модули `core/`

- **`core/protocol`**: единый формат сообщения (`protocol.Message`), сериализация и валидация.
- **`core/transport`**: TCP listen/send/accept/receive + утилита проверки порта.
- **`core/discovery`**: LAN discovery по UDP broadcast (анонсирует `peer_id` и TCP порт).
- **`core/chat`**: локальные “данные пользователя”:
  - контакты (contacts)
  - чёрный список (blocklist/ЧС)
- **`core/history`**: локальное хранение истории сообщений (JSONL по `chat_id`), дедуп по `message_id`, file-lock.
- **`core/messaging`**: небольшой слой поверх transport для сопоставления `addr -> peer_id` и inbox-канала (частично используется/задел).
- **`core/transport/p2p`**: экспериментальная папка (сейчас только `drop.go`).

### Поток данных: discovery

1) Каждый экземпляр периодически (раз в 2 секунды) посылает UDP broadcast на порт `DiscoveryPort`:

`SYNE1|<peer_id>|<tcp_port>`

2) Параллельно слушает UDP порт `DiscoveryPort` на `0.0.0.0`.

3) При получении валидного пакета вызывает callback `onPeer(peerID, "ip:port")`.

**Важно**: discovery не “подключает” чат сам по себе — он лишь даёт список соседей.

### Поток данных: сообщения (TCP + protocol)

Сообщение — это JSON (`protocol.Message`), который проходит:

- `protocol.ValidateMessage` на стороне отправителя (перед отправкой)
- `protocol.UnmarshalMessage` + `ValidateMessage` на стороне получателя

Далее логика делится на “для меня” и “для ретрансляции”:

- Если `TargetID == localID`, сообщение считается адресованным текущему пиру.
- Если `TargetID != localID`, CLI может форвардить сообщение “соседям” (контакты + LAN discovery) с уменьшением `ttl` (HopTTL).

**TTL** сейчас используется только для `MsgChat` и предотвращает бесконечные циклы ретрансляции.

### Контакты и ЧС (локальные файлы)

Контакты:

- путь: `data/contacts/contacts.jsonl`
- формат: JSONL (каждая строка — один `chat.Contact`)
- ключи “уникальности”: `peer_id` и `name` (без учёта регистра)

Чёрный список:

- путь: `data/blocklist/blocklist.jsonl`
- формат: JSONL (каждая строка — `chat.BlockedPeer`)
- блокировка основана на `peer_id`

### История сообщений

- путь: `data/history/<sha256(chat_id)>.jsonl`
- формат: JSONL (`history.StoredMessage`)
- дедупликация: если у сообщения есть `message_id`, то при сохранении проверяется наличие такого id в файле и второй раз не пишется
- конкурентный доступ: “файловый лок” через создание lock-файла `<history>.lock` (O_EXCL) с таймаутом

### Где находится “оркестрация”

Сейчас оркестрация в основном живёт в `cli/` (циклы accept/read, команды, логика форвардинга и т.п.). Core-пакеты намеренно маленькие и “плоские”: их проще поднимать в будущем UI/API слой.

### Планируемая эволюция

- **Transport**: перейти от “dial-per-message” к управляемым соединениям и/или поверхностному фреймингу.
- **Crypto**: подпись/шифрование payload, привязка `peer_id` к публичному ключу.
- **Messaging**: единый сервис маршрутизации/ретрансляции и контроль дубликатов/ACK на уровне core.
- **Storage**: переключение истории/контактов на SQLite (когда появится UI и миграции).

# Syne Core Architecture (MVP)

## 1. Scope

Этот документ фиксирует базовую сетевую модель ядра для MVP и служит источником истины для transport/messaging слоев.

## 2. Network Model

- 1 client process = 1 UDP socket = 1 local port.
- Количество чатов и групп не влияет на количество локальных портов.
- Порт выбирается при старте клиента (явно через `--port` или через auto-pick).

## 3. Identity and Addressing

- `peerID` - логический идентификатор участника.
- `addr` (`IP:port`) - текущая транспортная точка доставки.
- Отправка в сеть выполняется только по `addr`.
- Бизнес-логика работает через отображение `peerID -> addr`.
- `addr` может меняться, `peerID` остается стабильным в рамках сессии.

## 4. Protocol Routing Rules

Минимально обязательные поля сообщения:

- `From`
- `Type`
- `Target`
- `TargetID`
- `ChatID`
- `Payload`
- `Timestamp`

Маршрутизация входящих сообщений:

- Личка: `Target = Peer` и маршрутизация в чат по `ChatID`.
- Группа: `Target = Group` и маршрутизация по `TargetID` (group ID) и `ChatID`.

## 5. Transport vs Messaging Responsibilities

- `transport`: отправка/прием UDP-датаграмм по `addr`; не знает ничего о чатах и группах.
- `messaging`: разбор protocol message, валидация полей, роутинг по `ChatID/TargetID`, fan-out для групп.

## 6. Group Delivery Model

- У группы есть membership: `groupID -> []peerID`.
- При отправке в группу выполняется fan-out: одно логическое сообщение отправляется каждому участнику группы по его `addr`.
- Группа не имеет отдельного сетевого адреса.

## 7. Invariants (Must Hold)

- Нельзя создавать отдельный локальный порт под отдельный чат или группу.
- Нельзя подменять `peerID` транспортным адресом.
- Нельзя использовать `PeerAddr` как "адрес чата".
- Любая маршрутизация в приложении должна использовать `ChatID` и `TargetID`.

## 8. Manual Validation Checklist

- Два клиента запускаются на разных локальных портах.
- Личное сообщение (`Target = Peer`) доходит до конкретного peer.
- Групповое сообщение (`Target = Group`) доходит всем участникам membership.
- При занятом базовом порте клиент выбирает следующий свободный.
