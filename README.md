Идея:
Я разрабатываю анонимное приложение для общения, ориентированное на работу даже при отсутствии интернета. Проект представляет собой лёгкий мессенджер с поддержкой двух режимов подключения: peer-to-peer и client-server.

Основная цель — обеспечить возможность переписки и голосового общения в локальных сетях (например, в одной Wi-Fi сети, компьютерном клубе или учебном заведении) без необходимости доступа к глобальному интернету. Пользователи смогут автоматически находить друг друга внутри сети и обмениваться сообщениями напрямую.

Приложение делает упор на скорость работы, минимальное потребление ресурсов и небольшой размер клиента. Интерфейс и функциональность будут упрощены по сравнению с классическими мессенджерами — без каналов и сложных социальных механик, только быстрые личные и групповые коммуникации.

Система аккаунтов основана на анонимной идентификации через криптографические ключи и сквозное шифрование сообщений. Пользователь может войти без регистрации, указав только никнейм. Такие временные аккаунты автоматически удаляются через несколько дней, что позволяет использовать приложение для разового общения без хранения персональных данных.

Приложение не позиционируется как замена существующих мессенджеров, а предназначено для ситуаций ограниченного доступа к интернету, локального взаимодействия и приватного общения.


По стеку:
🖥 Frontend
Tauri + React + TypeScript
React — UI
Tauri — desktop runtime
Архитектура:
React → Tauri → Go Core
⚙️ Backend / Core
Go (net/http для MVP)
core/
 ├── transport/       # низкоуровневый обмен пакетами
 │    ├── tcp.go      # базовый UDP транспорт
 │    └── relay.go    # (опционально) relay соединения
 ├── discovery/       # поиск пиров в сети
 │    └── lan.go      # локальный поиск через UDP multicast
 ├── protocol/        # форматы сообщений, идентификаторы, версии
 │    └── messages.go
 ├── crypto/          # генерация ключей, шифрование, подпись
 │    └── keys.go
 ├── messaging/       # логика отправки/приёма сообщений, очередь
 │    └── messenger.go
 └── storage/         # SQLite или in-memory storage
      └── history.go
Абстрагирует P2P и client-server
UI вызывает только API core
🗄 Database
Server / relay / bootstrap nodes: PostgreSQL
Client: SQLite для локального кеша / истории
🐳 Инфраструктура
Docker — для relay / bootstrap / optional server nodes
Redis — позже, MVP 2.0, для онлайн-сессий и кэширования presence
⚡ Ключевой принцип:
Frontend ничего не знает о сети. Он просто вызывает core API, а Core решает, через P2P или сервер отправлять сообщение.
Такой стек спокойно позволит объединить per-to-per и client-server режимы, при этом код UI остаётся один.
